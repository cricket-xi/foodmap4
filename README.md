# **产品需求文档 (PRD): 大学城美食地图 (项目代号: 校园食集)**

**💡 执行摘要 (Executive Summary)** “校园食集”致力于成为大学城师生首选的美食决策平台。 面对美团/大众点评评价泛化、水军泛滥的痛点，我们主打 **“纯净的圈层信任 (TrustScore系统)”**；针对大学生高频的日常就餐选择困难症，我们提供 **“吃啥盲盒 (极致决策)”**。在商业化闭环上，我们为周边长尾中小商家提供 **“极简的一键式营销工具”**，从而构建“C端真实口碑沉淀 \-\> 流量分发 \-\> B端发券引流 \-\> 线下核销反哺C端粘性”的校园本地生活双边生态。

## **1\. 产品概述**

* **1.1 产品名称:** 校园食集 (Campus Foodie Map)  
* **1.2 产品定位:** 一款专为大学城师生打造的校园周边美食探索与决策辅助平台。  
* **1.3 产品形态:** 跨端 Web App / 微信小程序 (初期双端部署：C端学生 \+ B端商家 \+ O端平台运营)。  
* **1.4 适配终端:**  
  * **移动端 (Mobile):** 原生小程序体验，主打走动、找店、点餐、核销等高频单手操作场景。  
  * **PC/网页端 (Pad/Desktop):** 宽屏沉浸体验，仅限 O端运营后台及 C端宿舍场景下的深度攻略查阅。  
* **1.5 核心双边价值:**  
  * **To C (学生):** 打破信息差，消除决策内耗，提供**100%真实校园评价**，获取专属学生特惠。  
  * **To B (商家):** 精准获取高校垂直流量，提供**极低学习成本**的自营销工具，沉淀私域客群。

## **2\. 需求背景与竞品分析**

| 竞品/平台         | 核心优势                           | 劣势 (针对大学生痛点)                                  | 我们的差异化壁垒 (校园食集)                                  |
| :---------------- | :--------------------------------- | :----------------------------------------------------- | :----------------------------------------------------------- |
| **大众点评/美团** | 商家全、信息基础完善、有团购补贴。 | 评价泛化且水军多；榜单不具备“校园特色”；决策链路长。   | **纯净圈层信任:** 引入学生认证+TrustScore动态权重模型，主打真实红黑榜。 |
| **小红书**        | 图文/视频种草属性强，内容精美。    | 搜索效率低；网红店偏多，缺乏日常高频简餐的客观评价。   | **极致决策效率:** 通过“盲盒转盘”和“校园专属LBS地标”解决高频日常决策。 |
| **校园表白墙**    | 极具本地化，同学互动真实。         | 信息碎片化无法沉淀；缺乏检索功能；容易被商业广告刷屏。 | **结构化数据:** 将碎片化的校园口碑转化为结构化的榜单和地图标签，长期沉淀。 |

## **3\. 目标用户与使用场景**

* **C端 (核心):** 18-25岁在校大学生、教职工。**特征:** 对价格敏感、喜欢尝试新鲜事物、依赖手机导航、容易受同伴影响。  
* **B端 (商家):** 大学城周边3公里内餐饮门店老板。**特征:** 极度渴望学生客流、营销预算有限、缺乏复杂SaaS系统的操作能力。  
* **典型场景:**  
  * 宿舍几人聚餐，意见不合，打开“吃啥盲盒”一键抽选。  
  * 新生入学，查阅“X大南门必吃红榜”和“踩雷黑榜”防坑。  
  * 商家下午茶闲时，发布“仅限今日的10张1元奶茶券”引流并扫码核销。

## **4\. 核心业务流程**

* **盲盒决策到店流程:** 打开小程序 \-\> 授权位置 \-\> 点击“吃啥盲盒” \-\> 设置过滤条件 (如：不吃辣，\<20元) \-\> 动画转盘抽取 \-\> 展示结果卡片 \-\> 点击导航前往 \-\> 用餐完毕 \-\> 平台推送评价邀请 \-\> 完成评价获经验值。  
* **商家发券核销流程:** B端小程序登录 \-\> 点击“发布优惠” \-\> 选择模板设置库存 \-\> 上架成功 \-\> C端用户抢券存入卡包 \-\> 到店出示二维码 \-\> B端扫一扫 \-\> 校验成功，扣减库存 \-\> 记录对账单。  
* **红黑榜生成闭环:** 用户带图评价 \-\> 系统基于地理位置校验是否真实到店 \-\> 计入数据库 \-\> 离线算法每日凌晨计算 TrustScore \-\> 更新店铺红黑指数 \-\> 更新前台榜单缓存。

## **5\. 详细功能需求设计 (C端/B端/O端)**

### **5.1 C端核心功能 (学生端)**

1. **首页发现模块 (Home):**  
   * **金刚区:** 校园红黑榜、吃啥盲盒、学生特惠、特色品类。  
   * **快捷筛选:** 支持按具体大学大门 (如A校南门)、品类进行横向滚动筛选。  
   * **Feed流:** 餐厅卡片包含头图、店名、红黑榜评分、人均价格、距离当前位置、核心标签 (如: 阿姨手抖、加饭免费)。  
2. **校园特色榜单模块 (Leaderboards):**  
   * **闭眼入红榜:** 收录“红榜指数\>85%”且总有效评价数\>50条的店铺。  
   * **踩雷黑榜:** 收录“黑榜指数\>40%”的店铺，卡片外显最高赞的“避雷评价(带图)”。  
   * **场景特色榜:** 基于标签动态生成，如深夜食堂/罪恶夜宵榜、期末赶工平价外卖榜。  
3. **盲盒决策模块 (Randomizer):**  
   * **核心玩法:** 老虎机/轮盘动画随机抽取一家符合条件的餐厅。  
   * **智能避坑:** 开关“不吃辣/香菜”，输入自定义关键词强制排除。  
4. **地图与个人中心 (Map & Profile):**  
   * **地图导航:** 悬浮面板支持按距离、评分过滤。  
   * **吃货等级 (Gamification):** 签到、发布评价获取经验值。高等级可获得更高的“评价权重”(TrustScore加成)、线下折扣特权。

### **5.2 B端功能 (商家极简工作台)**

*坚持“极简、一键式操作”原则，极大降低数字化工具门槛，适配校园周边小吃街/档口老板的真实操作习惯。*

1. **极简入驻与门店管家:**  
   * **认领/新建POI:** 提供全平台模糊搜索认领或基于定位新建店铺。  
   * **资质极速审:** 手机拍照上传“营业执照+店主手持身份证”，系统 OCR 提取信息。  
   * **一键状态切换 (核心特性):** 首页提供红绿黄三色物理大按钮：🟢营业中 / 🔴已打烊 / 🟡爆满排队中 (预警)。当点击爆满时，C端学生主页会立刻出现“排队较久”的标签提示，起到分流预期管理作用。  
   * **基础信息维护:** 商家可一键修改营业时间、联系电话，并在主页发布“店铺小喇叭”（如：“阿姨今天老家有事，关门一天”）。  
2. **招牌菜与菜单管理:**  
   * 摒弃传统的全量菜单录入，主推\*\*“招牌菜管理”\*\*。商家只需上传店内最核心的 3-5 道必点菜图文和价格。  
   * **售罄开关:** 针对容易估清的热门菜品，支持一键点击“标记售罄”，实时同步到C端防空跑。  
3. **傻瓜式营销中心 (拉新引流):**  
   * **模板化发券:** 预设“引流秒杀款（如1元奶茶）”、“常规满减款（满20减2）”、“到店赠品款（送荷包蛋）”三大模板。商家仅需填入【面额】、【发放数量】即可发布，无需学习复杂的规则组合。  
   * **闲时特惠定向派发:** 鼓励商家在下午非饭点时段（如 14:00-16:00），发布限时专属券，系统会自动 Push 给关注了该店铺或品类的学生。  
4. **核销大厅与对账统计:**  
   * **全屏扫码台:** 打开即进入全屏扫码状态，支持调用手机后置摄像头扫学生券码，0.5秒极速核销。  
   * **极简每日对账:** 每天打烊时自动生成卡片式战报：“今日共引流 XX 人次，核销优惠券 XX 张，累计被查看 XX 次”。以直观的数据增强商家粘性。  
5. **真实客情维护与预警:**  
   * **差评即时预警:** 当收到黑榜评价或1星评价时，通过微信服务号模板消息第一时间推送给老板。  
   * **商家回复与申诉:** 提供快捷短语库回复评价。若遇到同行恶意抹黑（如虚假指控吃出虫子），商家享有\*\*“一键申诉”\*\*通道，可上传监控或沟通截图交由平台O端仲裁庭处理。

### **5.3 O端功能 (平台总控PC后台)**

*平台总控台是整个系统的“大脑”，支撑精细化运营、风控安全与数据决策。采用 RBAC（基于角色的权限控制）模型，支持平台管理员和“校园大使（按校区分权）”分级管理。*

1. **数据大盘与BI (Dashboard):**  
   * **核心看板:** 全局及单校区 DAU/MAU、新增注册用户数、活跃商家数趋势图。  
   * **业务转化漏斗:** 盲盒抽取 \-\> 点击导航转化率；发券量 \-\> 抢券量 \-\> 线下核销漏斗监控。  
   * **热力图分析:** 核心搜索词热力云图（洞察最近学生饮食趋势）、区域商户热力图。  
2. **用户管理中心 (User Management):**  
   * **C端用户管理:** 查看用户画像及活跃状态，支持封禁违规/刷单账号，人工调整用户吃货等级或违规扣分。  
   * **学生认证审核:** 对接学信网 API 自动校验或进行人工（学生证/校园卡图片）审核，处理认证异常申诉工单。  
3. **商户与POI管理 (Merchant Management):**  
   * **资质审核与入驻:** 审核B端上传的营业执照及店主身份信息，确保商家合规。  
   * **店铺数据治理:** 合并重复的地图 POI，修改错误纠偏坐标，管理店铺核心标签（如：清真、可包场、通宵营业）。  
   * **奖惩与干预:** 对存在食品安全隐患或严重刷单违规的店铺，可一键执行“强制下架”、“清空红榜指数”或冻结发券权限。  
4. **内容风控与仲裁庭 (Content & Risk Control):**  
   * **图文风控审核:** 接入 AI 机审+人工复审工作流，拦截C端发布的违禁词、低俗涉黄图片或涉嫌拉踩的恶意评价。  
   * **商家申诉仲裁:** 集中处理商家对“恶意差评”提起的申诉工单，客服核实后可操作“折叠评价”并对发布者降权。  
   * **敏感词词库管理:** 动态增删、维护平台违禁词、竞品词和引流黑词库。  
5. **营销与资源位运营 (Marketing & Ops):**  
   * **App/小程序装修:** 首页 Banner 轮播图配置、弹窗公告配置、金刚区图标动态替换。  
   * **场景榜单干预:** 灵活配置并发布“校园场景特色榜”（如：开学季迎新必吃榜、期末复习外卖榜），支持人工在榜单头部挂载推荐。  
   * **消息推送 (Push):** 按校区、按用户标签定向推送通知（例如：“你关注的烤肉店发1元特价券啦”）。

## **6\. 数据指标与埋点设计**

为了衡量产品健康度与商业化潜力，需定义以下核心数据指标：

* **北极星指标 (North Star Metric):** 周活跃学生用户数 (WAU) 与 盲盒抽取转化率。  
* **C端核心埋点:**  
  * 盲盒漏斗：点击盲盒按钮 \-\> 成功抽取 \-\> 点击去这里/导航 \-\> 实际产生评价。  
  * 发券漏斗：曝光量 \-\> 抢券成功率 \-\> 线下核销率 (关键闭环指标)。  
* **内容健康度:** 每日新增评价数、带图评价占比、恶意评价举报率。  
* **B端活跃度:** 商家周登录频次、主动发券商家占比。

### **7.2 项目技术架构与选型 (Technical Architecture)**

项目采用前后端分离、微服务/模块化的架构思想，优先保障移动端LBS性能和饭点期间的高并发处理能力。

1. **前端层 (Frontend Layer):**  
   * **框架选型:** 前端全面采用 Vue3，结合 Taro 或 Uni-app 编译至微信小程序和 Web 端。  
   * **地图与动画:** 接入腾讯地图 SDK；使用原生动效或基于 Vue3 的动画库实现顺滑的盲盒抽奖。  
2. **网关与接口层 (API Gateway Layer):**  
   * **网关选型:** Nginx \+ Spring Cloud Gateway。  
   * **安全限流机制:** 在网关层部署 Token 令牌桶算法，防止恶意脚本/黑客每秒发起上千次转盘抽取 (接口防刷)。  
3. **后端微服务层 (Backend Layer):**  
   * **语言选型:** 后端核心采用 Spring Boot 3。充分利用 Spring Boot 3 结合 JDK 21+ 虚拟线程 (Virtual Threads) 的高并发优势，完美契合饭点期间瞬间流量洪峰下的盲盒抽取与发券逻辑。  
   * **消息队列中间件:** 引入 Kafka 或 RabbitMQ，专门用于“用户点赞、经验值增加、抢券成功后的 DB 落盘”等非阻断式异步任务的削峰填谷。  
4. **数据与持久层 (Data Layer):**  
   * **关系型主库:** PostgreSQL \+ PostGIS (强空间地理数据计算能力)。  
   * **高并发缓存中心:** 深度整合 Redis。通过 Redis Cluster 承担 LBS 粗筛、防超卖 Lua 脚本执行、榜单缓存及分布式锁，与 Spring Boot 3 协同构建高并发护城河。

## **8\. 核心算法与高并发场景设计**

大学城场景的典型特征是\*\*“极端的饭点潮汐流量”\*\*。中午 11:30-12:30 期间会产生瞬时洪峰。

**8.1 盲盒加权随机抽取算法 (LBS高并发破局)**

* **痛点:** 距离计算+标签过滤+权重排序极易打满数据库CPU。  
* **方案 (Redis Geo \+ 内存精筛):**  
  1. 预热商家坐标及基础属性至 Redis Geo。  
  2. GEORADIUS 指令极速拉取 1km 内商户ID列表 (\<5ms)。  
  3. 应用层(Spring Boot内存)根据缓存剔除忌口词和黑榜标签。  
  4. 内存生成加权随机数，命中目标，全程不查 DB。

**8.2 优惠券秒杀防超卖算法 (Lua+MQ)**

* **痛点:** 1000人抢50张券引发数据库行锁竞争和超卖。  
* **方案:** Redis 存储库存 \-\> Lua 脚本保证原子性扣减 \-\> 异步投递 MQ (Kafka/RabbitMQ) \-\> 消费者平滑将发券记录写入 MySQL。

**8.3 TrustScore 权重与榜单生成算法 (异步离线)**

* **公式:** 权重(W) \= 基础分(认证状态) × 等级系数(Lv) × LBS真实度系数(\<500m为1.2，云评价为0.5)。  
* **防腐红榜率:** (∑红榜有效权重) / (∑红黑榜总权重)。  
* **高并发应对:** 采用定时任务(T+H或凌晨)离线计算，结果序列化入 Redis。C端刷新只读缓存，时间复杂度 O(1)。

## **9\. 社交裂变与分享机制**

* **神仙榜单分享:** 生成精美长图海报：“X大最新餐饮红黑榜出炉！”。  
* **求搭子卡片:** 盲盒结果分享：“我刚抽中了麦当劳，有人一起拼个饭吗？”。  
* **裂变拉新:** C端用户领券分享给好友，好友领取后，分享者获得经验值奖励。

## **10\. 异常与边界条件处理 (Exception Handling)**

1. **GPS定位被拒绝或获取失败:** 提示用户手动选择学校校区作为中心坐标。  
2. **盲盒条件过于苛刻 (找不到餐厅):** 弹出趣味提示：“要求太高啦，方圆一里连根草都没了！”，并建议用户放宽过滤条件。  
3. **网络极差 (校园网死角):** APP内预埋基础骨架屏 (Skeleton Screen) 和离线缓存的“校区常驻红榜”静态数据，保证无网也能看基本推荐。  
4. **防恶意刷榜机制 (防刷子):** \* 同一设备号 (DeviceID) 每日评价上限限制。  
   * 发布评价时校验 GPS 距离，距离商家超过1km的评价强制标记为“云评价”，不计入红黑榜核心权重。

## **11\. 法律合规与风险声明**

* **数据隐私 (PIPL):** 获取GPS需明确授权。学生证等敏感照片审核后执行“加水印脱敏”或“定时清除”。  
* **免责声明:** 注册页加入：“本平台不对商家食品安全、实际价格负责。”  
* **防恶意攻击:** 平台提供商家申诉入口；确凿恶意差评予以折叠，将恶意账号列入黑名单。

## **12\. 产品演进与商业化路线 (Roadmap)**

* **v1.0 (MVP期):** 基础LBS地图、盲盒、无门槛图文流（靠运营团队预埋数据做冷启动，暂不开放B端）。重点做“吃啥盲盒”单点突破。  
* **v2.0 (圈层护城河):** 上线学生认证与TrustScore，正式发布“红黑榜单”；上线B端，跑通发券核销全链路。  
* **v3.0 (商业化爆发):** 接入外卖CPS变现（点击跳美团/饿了么抽成）；允许商家购买“定向曝光券”或榜单“广告推荐位”。  
* **v4.0 (生态破圈):** 拓宽至“剧本杀、理发”等高频生活服务；承接全国快消品牌（如可口可乐、元气森林）校园开屏硬广。

## **13\. 非功能性需求与服务治理**

* **性能指标:** 移动端首屏 (FMP) \< 1.5s；盲盒抽取接口 99% 响应时间 (P99) \< 200ms。  
* **饭点自动降级策略 (Graceful Degradation):** 当系统负载过高 (如中午12:00)，自动触发降级开关：关闭UGC图片上传功能，延迟评价分数入库，系统资源全部让位给“找店/抽取盲盒/领券核销”三大主链路。  
* **防缓存雪崩:** 榜单和商户详情在 Redis 中设置随机过期时间；对不存在的恶意查询商户ID，在缓存中写入空对象防止穿透。
